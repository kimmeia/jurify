Diagnóstico

🐞 Bug do "campo personalizado"

O que você esperava

Cliente diz: "meu CPF é 123.456.789-00, nasci em 12/05/1985"

↓ IA entende sozinha ↓

Sistema salva cpf e dataNascimento no cadastro do cliente.

Mas hoje a IA só gera texto bruto — quem decide salvar é um nó manual definir_campo_personalizado com chave fixa e valor de variável. Não tem variável estruturada ({{cpf}}) porque a IA não retorna JSON.

A solução (novo nó)

IA EXTRAIR CAMPOS

Campos a extrair:

  • cpf (cpf)
  • dataNascimento (data)

Persistir: ✅ todos

{{extracao.cpf}} {{extracao.dataNascimento}}

IA recebe a mensagem + lista do que extrair + tipo de cada campo. Devolve um objeto estruturado via tool calling (suportado por Anthropic e OpenAI). Salva direto. Sem invenção.

Fundação

🧩 6 nós universais novos

Esses 6 blocos cobrem 99% dos cenários que você descreveu. Combinados, dão fluxos infinitos.

IA

ia_extrair_campos

IA lê uma mensagem + esquema de campos → devolve JSON estruturado. Salva no contexto e/ou em campos personalizados do cliente.

Recebe: mensagem, lista de campos com tipo
Publica: {{extracao.<chave>}} por campo
Tech: tool calling (Anthropic/OpenAI)

CRM

crm_buscar_contato

Busca contato por telefone, email ou CPF. Útil quando o telefone do gatilho não bate mas o cliente se identificou na conversa.

Recebe: tipo de busca + valor
Publica: contatoEncontrado, contatoId, nomeCliente, cliente.campos.*
Caso: "Cliente liga de número novo mas informa CPF"

CRM

crm_listar_acoes_cliente

Lista as ações (processos) do cliente. Filtros por tipo, polo, limite. Resultado vai no contexto pra iterar ou exibir.

Recebe: contatoId, filtros
Publica: {{acoes}} (lista), {{acoesQuantidade}}
Caso: "Quantas ações esse cliente tem?"

PROCESSOS

processo_buscar_movimentacoes

Histórico do processo: movimentações, publicações DJE, sentenças, audiências. Filtros por tipo e janela (últimos N dias / N eventos).

Recebe: processoId ou CNJ + filtros
Publica: {{movimentacoes}}, {{movimentacaoMaisRecente}}
Caso: "Houve novidade no meu processo?"

MENSAGEM

whatsapp_aguardar_resposta

Envia mensagem e pausa o fluxo aguardando o cliente responder. Timeout configurável (default 24h). Permite conversa multi-turn.

Recebe: mensagem + timeout
Publica: {{respostaUsuario}}
Caso: "Qual horário prefere? → espera resposta → confirma"

FLUXO

para_cada_item

Loop sobre lista do contexto (ex: cada ação do cliente). Executa subfluxo pra cada, com {{item}} e {{indice}} disponíveis.

Recebe: caminho da lista + limite
Saídas: 2 handles (corpo do loop + depois do loop)
Caso: "Pra cada ação → cria card no Kanban"

🎯 Por que isso destrava infinitos cenários

  • Os mesmos nós podem ser recombinados pra fluxos completamente diferentes
  • Aguardar resposta permite conversas multi-turn (ida-e-volta) — não só "IA responde e fim"
  • Para cada + qualquer nó vira processamento em lote
  • Buscar contato/ações/movimentações dá inteligência real — IA não inventa, lê do banco
  • Extrair campos com tool calling = IA estruturada, não texto bruto

❓ Antes de eu codar

Pergunta 1

O bug do campo personalizado era esse mesmo?

Hipótese: você queria que a IA salvasse sozinha, mas hoje precisa de passo manual. ia_extrair_campos resolve. Se era outro caso, me diz qual.

Pergunta 2

Os 6 nós cobrem seus cenários?

Pensei em casos genéricos. Tem algum caso específico seu que ainda não tá coberto? Ex: "IA precisa X ação que não vi aqui".

Pergunta 3

Tool calling agora ou depois?

Com tool calling = IA estruturada confiável. Sem tool calling = JSON-mode (funciona mas inventa às vezes). Recomendo COM — pouca coisa a mais.

Pergunta 4

Visual do nó: ok ou ajustes?

Header colorido + preview + chips. Pode ficar muito ocupado se nó tem muita config. Quer manter assim ou simplificar?

Plano de execução em 4 PRs (~1 dia cada)

  1. PR-4 — ia_extrair_campos + tool calling + avisos visuais quando config inconsistente. Resolve o bug.
  2. PR-5 — crm_buscar_contato, crm_listar_acoes_cliente, processo_buscar_movimentacoes.
  3. PR-6 — whatsapp_aguardar_resposta, para_cada_item, menu de opções automático.
  4. PR-7 — Visual repaginado dos nós + tooltips ricos + validação inline + wizard primeiro fluxo.

Cada PR roda pnpm check + test verde antes de seguir. Motor existente não muda (zero regressão garantida).
